# FinPilot AI — Personal Finance Operating System (PWA)
## Final Production Prompt

---

## Objective

Build a premium Progressive Web App (PWA) for Indian salaried individuals to manage:

- Monthly cash flow
- Expenses
- Investments
- Emergency fund
- Home loan prepayments
- Net worth
- Financial planning and recommendations

The app should answer one simple question every month:

> "Based on my salary and expenses, how much should I invest, keep for emergencies, and prepay towards my home loan?"

The app must be installable on iPhone and Android as a PWA and hosted on GitHub Pages.

## Rule-Based Financial Advisor Engine

NOT an AI/LLM. This is a deterministic rule engine. Do not label it "AI" anywhere in the UI.

Name it: "Smart Planner" or "Financial Planner".

Input variables:
- Monthly net income (after deductions)
- Total monthly expenses (recurring + one-time)
- Current emergency corpus
- Emergency corpus target (monthly expenses × target months)
- Current investments
- Home loan outstanding, rate, tenure
- Monthly surplus
- Risk profile

Priority-ordered allocation rules:

1. **Essential Expenses** — always funded first (recurring + one-time)
2. **Emergency Fund** — if corpus < target, allocate min(surplus × 0.3, shortfall) to emergency fund
3. **Investments (SIP)** — allocate based on risk profile:
   - Conservative: 10–15% of net income
   - Balanced: 15–25% of net income
   - Aggressive: 25–35% of net income
4. **Home Loan Prepayment** — if surplus remains after above, suggest prepayment amount
5. **Buffer** — retain minimum ₹5,000 or 3% of income as discretionary buffer

Allow the user to customize the priority order in Settings.

Recommendation messages must be plain English. Examples:
- "Your emergency fund covers only 4 months. Target is 9 months. Contributing ₹10,000 this month."
- "At current prepayment pace, you'll be debt-free by March 2031, saving ₹4.2L in interest."
- "Your SIP of ₹20,000 is on track for your retirement goal."

## Auto Home Loan Updater

### SBI Daily Reducing Balance — Exact Formula

SBI calculates interest on a **daily reducing balance** basis, NOT monthly compounding. The formula is:

```
Interest for a period = Outstanding Amount × Rate × Number of Days / 36500
(use 36600 if the period spans a leap year's Feb 29)
```

The EMI is fixed. When the interest rate changes on a floating rate loan, **the EMI stays the same but the tenure adjusts** — never change the EMI amount.

### Rate Change History Storage

The `loanSetup` document must store the full rate history as an array. Example:

```json
{
  "loanName": "SBI Home Loan",
  "originalAmount": 7400000,
  "disbursementDate": "2023-04-20",
  "emi": 61000,
  "rateType": "floating",
  "rateHistory": [
    { "effectiveDate": "2023-04-20", "rate": 8.70 },
    { "effectiveDate": "2025-02-15", "rate": 8.45 },
    { "effectiveDate": "2025-04-15", "rate": 8.20 },
    { "effectiveDate": "2025-06-15", "rate": 7.70 },
    { "effectiveDate": "2025-12-15", "rate": 7.45 }
  ],
  "currentRate": 7.45
}
```

The UI in Loan Settings must allow the user to:
- Add a new rate change entry (date + new rate)
- Edit or delete past rate change entries
- The system recalculates everything from disbursement date forward on any change

### Amortization Engine (loanCalculator.ts)

The amortization schedule must be built by **replaying every day from disbursement date to loan closure**, applying the correct rate from `rateHistory` on each day:

```typescript
function buildDailyAmortization(loan: LoanSetup, prepayments: LoanHistory[]): AmortizationResult {
  let outstanding = loan.originalAmount;
  let date = new Date(loan.disbursementDate); // e.g. 2023-04-20
  const emiDay = date.getDate();              // day of month EMI is due (e.g. 20)
  let totalInterestPaid = 0;
  let monthlySchedule = [];

  while (outstanding > 0) {
    const monthStart = new Date(date);
    const nextMonth = new Date(date.getFullYear(), date.getMonth() + 1, emiDay);
    const days = daysBetween(date, nextMonth); // actual calendar days in this period
    const isLeap = isLeapYear(date.getFullYear());
    const divisor = isLeap && spansLeapDay(date, nextMonth) ? 36600 : 36500;
    const rate = getRateForDate(loan.rateHistory, date); // look up active rate

    const interestAccrued = outstanding * rate * days / divisor;
    const principalPaid = Math.min(loan.emi - interestAccrued, outstanding);

    // Check if any prepayment falls in this month
    const prepaymentThisMonth = getPrepaymentForMonth(prepayments, date);

    outstanding = outstanding - principalPaid - prepaymentThisMonth;
    if (outstanding < 0) outstanding = 0;

    totalInterestPaid += interestAccrued;
    monthlySchedule.push({
      month: formatYYYYMM(date),
      openingBalance: outstanding + principalPaid + prepaymentThisMonth,
      emi: loan.emi,
      interestComponent: interestAccrued,
      principalComponent: principalPaid,
      prepayment: prepaymentThisMonth,
      closingBalance: outstanding,
      rate
    });

    date = nextMonth;
  }

  return { schedule: monthlySchedule, totalInterestPaid, closureDate: date };
}
```

Key rules:
- `daysBetween` must count **actual calendar days** (not assume 30 days/month)
- The rate lookup must find the most recent `rateHistory` entry whose `effectiveDate <= current date`
- Prepayments reduce the outstanding from the day they are applied, which shortens tenure
- If `EMI < interestAccrued` in any month (rate rose too high), flag this as a negative amortization warning

### After Each Monthly Entry

When the user records a prepayment in the Monthly Planner, log to `users/{uid}/loanHistory/{id}`:

```json
{
  "date": "2026-06-20",
  "prepaymentAmount": 50000,
  "outstandingBefore": 5800000,
  "outstandingAfter": 5750000,
  "tenureReducedByMonths": 3,
  "cumulativeInterestSaved": 142000
}
```

Recalculate the full amortization from disbursement date on every prepayment entry. Derive `outstandingAmount` and `remainingTenureMonths` from the schedule rather than storing them as raw editable fields — they are always computed outputs.

Display all updated values (closure date, interest saved, new tenure) immediately after entry. No manual calculation required from the user.
